1.1.1 LangChain核心模块划分

LangChain的核心组件，从逻辑上可以划分为以下四大部分：Model I/O、Chains、RAG、Agents。



1）Model I/O

标准化大模型的输入和输出，包含提示模版，模型调用和格式化输出。

（1）Format（格式化）：通过模板管理大模型的输入。将原始数据格式化成模型可以处理的形式，插入到一个模板中，然后送入模型进行处理。

（2）Predict（预测）：调用 LLM 接收输入，进行预测或生成回答。

（3）Parse（解析）：规范化模型输出。比如将模型输出格式化为 JSON。

2）Chains

“链条”用于将多个组件组合成一个完整的流程，方便链式调用。



3）Retrieval

对应RAG：检索外部数据，作为参考信息输入LLM辅助生成答案。

1.1.1 RunnableParallel 可运行并行

RunnableParallel 同时运行多个可运行对象，并为每个对象提供相同的输入。

对于同步执行，RunnableParallel 使用 ThreadPoolExecutor 来同时运行可运行对象。对于异步执行，RunnableParallel 使用 asyncio.gather 来同时运行可运行对象。

构造RunnableParallel实例时，参数列表是可变数量关键字参数，一个参数名对应着一个可运行组件，每个可运行组件输出结果将作为参数名key所对应的值，封装到整个运行实例的结果当中。

### 一、 LCEL 与基础概念

**1. 什么是 LCEL？为什么要用它来构建链？它解决了什么问题？**

- **答题思路**：
  - **定义**：LCEL 是 LangChain 表达式语言（LangChain Expression Language）。它是一种**声明式**的链式组合方式。
  - **核心价值**：所有由 LCEL 构建的组件都实现了统一的 **`Runnable` 接口**。这意味着它们天然支持 `invoke`、`batch`、`stream` 等方法。
  - **解决的问题**：传统方式构建链需要继承复杂的 `Chain` 类，代码冗长。LCEL 通过 `|` 管道操作符，让组合变得像写 shell 命令一样简洁直观。

**2. LangChain 中的 `Runnable` 接口是什么？它和传统的 `Chain` 有什么区别？**

- **答题思路**：
  - **`Runnable` 接口**：这是 LangChain 的**标准执行接口**，被模型、解析器、检索器等绝大多数组件实现。它提供了一致的 `invoke`/`ainvoke`（调用）、`batch`/`abatch`（批处理）、`stream`/`astream`（流式）等 API。
  - **与 `Chain` 的区别**：`Runnable` 是更底层、更统一的抽象。传统的 `Chain` 是旧版 API，功能相对单一，而实现了 `Runnable` 的 LCEL 链具备更强的组合性和更丰富的执行能力。

**3. 请写出一个使用 LCEL 的最小化链，它包含 Prompt 模板、模型和输出解析器。**

- **答题思路**：展示一个标准的三段式链 `prompt | model | output_parser`。

  python

  ```
  from langchain_core.prompts import ChatPromptTemplate
  from langchain_openai import ChatOpenAI
  from langchain_core.output_parsers import StrOutputParser
  
  # 1. 定义提示词模板
  prompt = ChatPromptTemplate.from_messages([
      ("system", "你是一个{role}专家。"),
      ("user", "{input}")
  ])
  # 2. 定义模型
  model = ChatOpenAI(model="gpt-4o-mini")
  # 3. 定义输出解析器
  parser = StrOutputParser()
  
  # 4. 使用 | 构建链
  chain = prompt | model | parser
  
  # 5. 调用链
  result = chain.invoke({"role": "Python", "input": "什么是装饰器？"})
  print(result)
  ```

### 二、 链（Chain）的类型与流程控制

**4. 什么是“顺序链”（Sequential Chain）？请举例说明其应用场景。**

- **答题思路**：
  - **定义**：顺序链是将多个链或组件按**固定的顺序**连接，前一个的输出自动成为后一个的输入。
  - **应用场景**：适用于步骤明确、流程固定的多步任务。例如：
    - **翻译与总结**：先将一篇英文文章翻译成中文，再对中文内容进行总结。
    - **大纲生成**：先根据主题生成文章大纲，再根据大纲生成完整文章。
  - 在 LCEL 中，顺序链就是简单地用 `|` 将多个 `Runnable` 连接起来。

**5. 什么是“路由链”（Routing Chain）？它解决了什么问题？**

- **答题思路**：
  - **定义**：路由链能够根据输入的内容或中间结果，**动态地选择**下一步要执行的链或工具。
  - **解决的问题**：它让应用具备了**决策能力**。例如，可以构建一个智能客服，如果用户问的是技术问题，就路由到技术文档链；如果问的是订单问题，就路由到订单查询链。这比固定的顺序链更灵活。

**6. 什么是“并行链”（Parallel Chain）？在什么场景下会用到它？**

- **答题思路**：
  - **定义**：并行链可以同时执行多个独立的 `Runnable`，最后将各自的结果聚合起来。
  - **应用场景**：当有多个相互独立、可以同时进行的任务时。例如：
    - **多源信息检索**：同时从向量数据库和搜索引擎中检索与问题相关的信息。
    - **多维度分析**：对一段文本同时进行情感分析、实体识别和关键词提取。
  - 在 LCEL 中，可以使用 `RunnableParallel` 来实现。

### 三、 组件集成与高级应用

**7. 如何在一个链中集成“记忆”（Memory）功能，以实现多轮对话？**

- **答题思路**：
  - **问题本质**：大模型本身是无状态的，无法记住之前的对话。
  - **解决方案**：LangChain 的 `Memory` 模块用于在链的多次调用之间持久化状态。
  - **实现方式**：将 `Memory` 组件（如 `ConversationBufferMemory`）作为一个变量，注入到 Prompt 模板中，并在每次调用链时更新和读取这个记忆。
  - **面试话术**：“我会使用 LangChain 的 `Memory` 组件，比如 `ConversationBufferMemory` 来存储对话历史。然后在构建 Prompt 时，将历史信息作为上下文的一部分传给模型，从而让模型‘记住’之前聊过的内容。”

**8. 请描述在 LangChain 中实现 RAG（检索增强生成）的核心流程。**

- **答题思路**：将流程分为**索引**和**查询**两个阶段。
  - **索引阶段（离线）**：
    1. **加载**：使用 `Document Loaders` 加载文档。
    2. **切分**：使用 `Text Splitters` 将长文档切分成小块。
    3. **向量化与存储**：使用 `Embeddings` 模型将文本块转为向量，并存入 `Vector Store`。
  - **查询阶段（在线）**：
    1. **检索**：将用户问题向量化，从 `Vector Store` 中检索出最相关的文本块。
    2. **增强**：将检索到的文本块作为上下文，与用户问题一起填入 `Prompt` 模板。
    3. **生成**：将完整的 Prompt 传给 `LLM` 生成回答。

**9. LangChain 中的“链”（Chain）和“代理”（Agent）有什么区别？**

- **答题思路**：
  - **链（Chain）**：是**固定的、预设的**执行流程。数据像在流水线上一样，按预先定义好的步骤流动。适合流程明确的任务。
  - **代理（Agent）**：是**动态的、自主决策的**。它由 LLM 作为“大脑”，根据用户输入，自主决定“要不要调用工具”、“调用哪个工具”、“传入什么参数”。适合流程不固定、需要灵活应变的任务。
  - **一句话总结**：**Chain 是按剧本演戏的演员，Agent 是能随机应变的导演。**

**10. 在 LangChain 中如何定义和使用“工具”（Tool），让模型能够调用外部 API 或函数？**

- **答题思路**：
  - **工具的本质**：工具是封装了外部功能的函数，并配有名称和描述，供模型调用。
  - **定义方式**：可以使用 `@tool` 装饰器将一个 Python 函数快速声明为一个工具，或者使用 `Tool` 类进行更复杂的定义。
  - **使用流程**：将定义好的工具列表传给 `Agent`。Agent 在收到用户请求后，会决策是否需要调用工具，如果需要，则生成调用指令并执行。

### 四、 性能、可靠性与部署

**11. 基于 LCEL 构建的链如何实现流式输出（Streaming）？**

- **答题思路**：
  - 基于 LCEL 构建的链（即所有实现了 `Runnable` 接口的组件）**自动**支持 `stream` 和 `astream` 方法。
  - 可以直接在链上调用 `.stream()` 方法来逐块获取输出。这能极大提升用户体验，让用户感觉模型在“实时思考”和输出。
  - *注意*：最终能否实现逐 token 流式输出，还取决于底层模型提供商是否支持。

**12. 如何实现链的并行处理和批处理（Batch Processing）？**

- **答题思路**：
  - 因为 LCEL 组件都实现了 `Runnable` 接口，所以它们都统一支持 `batch` 和 `abatch` 方法。
  - 这意味着可以传入一个输入列表，让链一次性处理多条数据，从而大幅提升吞吐量，而无需为每个组件编写不同的并发代码。

**13. 如何对链的执行过程进行监控和调试？**

- **答题思路**：
  - **LangSmith**：这是 LangChain 官方提供的商业化可观测性平台。通过配置环境变量，所有链的执行过程（包括输入、输出、延迟、Token 消耗等）都会自动被记录和可视化。
  - **开源替代方案**：也可以集成社区的开源方案，如 **Langfuse**，来实现类似的追踪和指标监控功能。

**14. 如何将构建好的链（`Runnable`）部署成一个 API 服务？**

- **答题思路**：
  - **LangServe**：这是 LangChain 官方提供的工具，专门用于将 `Runnable` 或链部署为 REST API 服务。它使用简单，并能自动生成 API 文档（类似 FastAPI）。

### 五、 场景设计题（高频）

**15. 面试官：假设我们要构建一个智能客服机器人，它需要能回答基于公司内部文档的问题，并且能查询订单状态。请描述你会如何使用 LangChain 的链或代理来实现？**

- **答题思路**：这是一个典型的 **RAG + 工具调用** 的场景，适合使用 **Agent**。
  1. **定义工具**：创建两个工具。一个是 `retrieve_from_knowledge_base`（从向量数据库检索公司文档），另一个是 `query_order_status`（调用订单 API 查询状态）。
  2. **初始化 Agent**：将这两个工具和 LLM 一起，初始化一个 ReAct 风格的 Agent。
  3. **工作流程**：当用户问“我的订单 #12345 到哪了？”时，Agent 会决策调用 `query_order_status` 工具。当用户问“你们公司的退货政策是什么？”时，Agent 会决策调用 `retrieve_from_knowledge_base` 工具。
  4. **优势**：这种方式比固定的链更灵活，可以应对用户各种混合型的问题。

1. 基础必考题：四种消息格式的底层原理
问题：LangChain支持字符串、元组、字典和对象四种格式传入消息，底层是如何统一处理的？哪种性能最好？

💡 回答思路：

核心机制：无论传入哪种格式，ChatOpenAI 底层都会调用 _convert_messages_to_openai_format() 这类转换器，将它们统一标准化为 OpenAI API 所需的 List[Dict] 格式（包含 role 和 content）。

效率差异：消息对象（Object）性能最好（省去类型推断），字符串性能最差（需额外解析）。因为传入字符串时，LangChain 必须执行类型推断（默认当成 HumanMessage），而传入对象时直接映射属性，减少了 isinstance() 和 role 映射的开销。

元组 vs 字典：元组解包速度略快于字典哈希查找，但差异微乎其微；字典可读性更好且兼容 OpenAI 原生 SDK。

2. 角色边界题：SystemMessage 的位置和覆盖规则
问题：在 input_message_object 中，SystemMessage 放在列表开头。如果我把 SystemMessage 放在列表中间或末尾，会发生什么？多条 SystemMessage 会怎样？

💡 回答思路：

位置规则：在 OpenAI 协议中，System prompt 必须位于对话列表的最前面（index 0）。如果放在中间或末尾，OpenAI 网关会报错（如 Invalid role: system）或直接忽略后续的 System 角色。

多条合并：如果传入多个 SystemMessage，LangChain 默认不会自动合并，而是直接透传给 API。但 OpenAI 服务端会报错（只允许一条 system）。实际开发中，建议手动拼接成一个完整的 SystemMessage，或使用 SystemMessagePromptTemplate 合并。

进阶点：可以提一下在 invoke 时传入的 config 中的 system 参数，会覆盖消息列表中的 SystemMessage。

3. 上下文陷阱题：AIMessage 在输入中的作用
问题：代码中把 AIMessage(content="汪汪！") 放在了用户输入中，这种做法有什么实际意义？如果不传历史 AIMessage，模型能记住刚才自己说过的话吗？

💡 回答思路：

核心作用：AIMessage 不是给模型“看”的，而是给模型“补全上下文”用的。因为大模型本身是无状态的，传入历史中的 AIMessage 是为了告诉模型“你刚才已经回复过‘汪汪’了”，从而实现多轮对话的连贯性（比如让模型接着“被踹”的情节继续演）。

无状态本质：如果不传入历史 AIMessage，模型每次调用都是独立的，它绝对记不住自己上一条输出。所以工程上必须由开发者维护整个 messages 列表，把每次的响应 AIMessage 追加进去，再传给下一次 invoke。

面试加分项：可以提到 RunnableWithMessageHistory 这类封装类，它能自动管理会话历史，避免手动维护列表的麻烦。

4. 高阶源码题：invoke 与 stream 的区别，以及异步处理
问题：llm.invoke() 是阻塞式调用。如果我想实现打字机流式输出，代码如何改动？四种消息格式在流式场景下有区别吗？

💡 回答思路：

改动方法：将 invoke 改为 stream，并用 for chunk in llm.stream(messages): print(chunk.content, end="")。

格式兼容：四种格式完全通用，因为流式调用发生在消息格式转换之后，底层只是将 generate 改为流式迭代器，对上层消息结构透明。

注意点：提到 AIMessageChunk，它是流式输出的增量片段，最终会聚合成完整的 AIMessage。异步场景则使用 ainvoke 和 astream。

5. 实战场景题：函数调用（Tool Calling）时的消息构造
问题：如果这段代码需要结合 Tool Calling（工具调用），消息列表要如何调整？

💡 回答思路：

新增角色：需要引入 ToolMessage（角色为 tool）。

消息顺序：System -> User -> AIMessage（带 tool_calls） -> ToolMessage（工具返回结果） -> AIMessage（最终回复）。

对应格式：如果使用字典格式，role 必须为 "tool"，且必须包含 tool_call_id 字段；如果使用对象格式，必须传入 ToolMessage(content="结果", tool_call_id="xxx")。

面试潜台词：面试官在考察你是否理解消息结构对 Agent 工作流的决定性影响。

💼 面试官终极追问（压轴题）
追问：既然有四种方式，你在生产环境中会选哪种？为什么？

✅ 标准答案：
生产环境强制使用 消息对象（Object）。
原因有三：

类型安全：配合 Pydantic 或 Typing，IDE 有智能提示，杜绝手写 "system" 拼写错误。

扩展性：AIMessage 可以携带 tool_calls 和 usage_metadata 等额外属性，而纯字典/元组放不下这些复杂结构。

链式兼容：LangChain 的 LCEL（如 | 管道符）要求传入的是 BaseMessage 子类，传字符串会破坏链式调用的类型推断。

-----------------------------------------------------------------------------------------------------------------

LangChain 的链式调用（Chaining），核心思想是将多个处理步骤通过管道操作符 | 连接起来，形成一个完整的数据处理流水线。它的本质是将多个组件（如模型、提示词模板、工具等）组合成一个可复用、可组合的工作流，让前一步的输出自动成为后一步的输入。

💡 核心概念：LCEL (LangChain 表达式语言)
在 LangChain 的现代用法中，LCEL 是构建链的首选和推荐方式。

| 管道操作符：这是 LCEL 的核心。它像 Unix 管道一样，将不同的“可运行对象”（Runnables）串联起来。

函数式组合：代码更简洁、直观，易于理解和维护。

最基本的链可以简单到只包含一个提示词模板和一个模型：

python
from langchain.prompts import PromptTemplate
from langchain_openai import ChatOpenAI

prompt = PromptTemplate.from_template("描述一下在{city}完美的一天。")
model = ChatOpenAI(model="gpt-4o-mini")

# 使用 | 运算符将提示词和模型组合成一个链
chain = prompt | model

# 调用链，传入输入
chain.invoke({"city": "巴黎"})
在这个例子中，city 参数首先被填入 prompt 模板，生成完整的提示词，然后传递给 model 得到最终回复。

常见的链类型
LangChain 支持构建不同复杂度的链来应对各种场景。

链类型	描述	适用场景
基础链 (Basic Chain)	最简单的形式，通常只包含一个提示词模板和一个模型调用。	简单的问答、文本生成等单一任务。
顺序链 (Sequential Chain)	将多个链按固定顺序连接，前一个的输出作为后一个的输入。	多步骤任务，如：先生成文章大纲，再根据大纲写全文；或者先总结文本，再翻译总结。
路由链 (Routing Chain)	根据输入内容或中间结果，动态选择下一步要执行的链或工具。	复杂的决策系统，如：根据问题的类型（技术/通用）调用不同的专家模型来回答。
并行链 (Parallel Chain)	同时调用多个工具或模型，最后将各自的结果聚合起来。	多模态分析，如同时分析一段文本的情感和一张图片的内容。
🛠️ 如何构建和调用链
1. 构建链的两种方式
现代方式 (推荐)：使用 LCEL
这是当前的主流实践，核心就是使用 | 管道操作符。

python
from langchain.prompts import PromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

model = ChatOpenAI()
prompt = PromptTemplate.from_template("讲一个关于{topic}的笑话")
output_parser = StrOutputParser() # 用于解析模型输出

# 用 | 组合成链
chain = prompt | model | output_parser
chain.invoke({"topic": "程序员"})
传统方式 (遗留)：继承 Chain 类
这是 LangChain 早期的用法，现在已不推荐，但在一些旧代码中可能还会见到。它通过创建类并实现 _call 方法来定义链的逻辑。

2. 调用链的方法
构建好的链可以通过以下方法执行：

invoke (推荐)：这是 LCEL 的标准调用方法，输入输出均为字典，功能最强大。

__call__：链是可调用的，__call__ 是执行链的主要方式，输入是字典，返回也是字典。

run：这是一个便捷方法，仅适用于单个输入和单个输出的链，直接返回字符串结果。

python
# 假设 chain 只有一个输入 'question' 和一个输出
result = chain.run("上海今天的天气怎么样？")[reference:24]
核心优势与最佳实践
核心优势：链式调用可以将复杂任务逻辑解耦与复用，并支持动态流程控制，如条件分支和循环。

最佳实践：

优先使用 LCEL：更简洁、灵活，是官方推荐的现代写法。

善用数据转换：在链的中间步骤使用普通 Python 函数或 RunnableLambda 来清洗、映射或转换数据格式。

管理状态与上下文：对于需要记忆的对话等场景，可以利用 Memory 模块在链的步骤间传递对话历史等状态信息。

LangChain 的链式调用提供了一套强大而灵活的模式，从简单的 API 调用到复杂的多步推理，都能通过组合不同的“零件”来优雅实现。

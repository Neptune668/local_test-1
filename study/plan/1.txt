### 1. 环境准备

- 使用 `dotenv` 加载环境变量（通常包含 `OPENAI_API_KEY`）。
- 实例化 `ChatOpenAI`，指定模型（示例中为 `'gpt-5.5'`，实际应替换为有效模型名，如 `gpt-3.5-turbo`）。

------

### 2. 四种消息列表格式

代码分别构造了四种格式的消息列表，并（注释掉部分）分别调用 `llm.invoke()` 获得响应。

| 格式             | 示例结构                                                     | 特点                                                         |
| :--------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **字符串**       | `"评价一下尚硅谷"`                                           | 最简单，直接传入用户提问字符串。LangChain 会自动将其视为 `HumanMessage`。 |
| **元组列表**     | `[("system", "你是一个美国人..."), ("user", "你好...")]`     | 每个元组由 `(角色, 内容)` 组成，角色可为 `"system"`、`"user"`、`"assistant"`。轻量，适合快速构造。 |
| **字典列表**     | `[{"role":"system","content":"..."}, {"role":"user","content":"..."}]` | 与元组类似，但用字典明确键名，可读性更好，兼容 OpenAI 官方格式。 |
| **消息对象列表** | `[SystemMessage(content="..."), HumanMessage(content="..."), AIMessage(content="..."), ...]` | 使用 LangChain 提供的消息类（`SystemMessage`、`HumanMessage`、`AIMessage`），支持完整的多轮对话历史（包含 AI 的中间回复）。功能最强，类型安全。 |

------

### 3. 调用与输出

- 所有格式最终都会由 LangChain 内部转换为统一的 `BaseMessage` 序列，再发送给模型。
- 调用 `llm.invoke()` 返回一个 `AIMessage` 对象，通过 `.content` 获取文本回复。
- 代码最后取消注释，使用**消息对象**格式（包含系统角色、用户、AI 中间回复和新的用户输入）进行调用，并打印模型的最终响应内容。

------

### 4. 关键点

- 四种格式在功能上等价，开发者可根据场景选择最方便的一种。
- **消息对象**格式最灵活，特别适合需要记录完整对话上下文（包括 AI 的历史输出）的多轮交互。
- 角色名称在元组/字典中需遵循约定（`system`/`user`/`assistant`），而消息对象类名直接对应角色。

------

### 5. 使用建议

- 简单单轮问答 → 字符串或元组列表。
- 多轮对话或需要精细控制角色 → 消息对象列表。
- 若与外部 API 交互且已有 OpenAI 格式数据 → 字典列表最直接。

这段代码很好地展示了 LangChain 在消息构建上的灵活性和易用性。 🚀
